# Feature Specification: HDC Plus - OpenHarmony 设备管理桌面应用

**Feature Branch**: `002-hdc-plus`

**Created**: 2026-07-02

**Status**: Draft

**Input**: User description: "基于 Tauri v2 + Vue 3 构建的 OpenHarmony 设备管理桌面应用，通过 HDC（OpenHarmony Device Connector）实现画面投屏、HAP 包安装、设备管理等功能。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 设备连接与管理 (Priority: P1)

作为一名 OpenHarmony 应用开发者，我希望能够快速发现并连接本地及网络中的 OpenHarmony 设备，查看设备的基本信息与状态，以便在开发测试过程中高效管理多台设备。

**Why this priority**: 设备连接是所有后续操作的前提，没有任何可管理的设备，投屏、安装等功能均无法使用。这是整个应用的基础能力。

**Independent Test**: 可以通过 USB 或局域网搜索 OpenHarmony 设备，成功连接后查看设备列表和设备基本信息，即验证了核心价值——设备可发现、可连接、可管理。

**Acceptance Scenarios**:

1. **Given** 应用已启动且电脑通过 USB 连接了一台 OpenHarmony 设备，**When** 用户打开应用，**Then** 系统自动检测到该设备并显示在设备列表中，包含设备名称、型号和连接状态。
2. **Given** 设备列表中有已连接的设备，**When** 用户断开设别连接（拔掉 USB 或网络断开），**Then** 设备状态实时更新为"已断开"，并在列表中给出视觉提示（如置灰）。
3. **Given** 电脑与 OpenHarmony 设备处于同一局域网，**When** 用户手动输入设备的 IP 地址进行连接，**Then** 系统通过网络成功建立 HDC 连接，设备出现在列表中并标记为"网络连接"。
4. **Given** 设备列表中有一台已连接的设备，**When** 用户点击该设备查看详情，**Then** 系统展示设备的操作系统版本、SN 序列号、存储空间使用情况等完整信息。

---

### User Story 2 - HAP 包安装与管理 (Priority: P1)

作为一名 OpenHarmony 应用开发者，我希望能够通过桌面应用将 HAP 安装包便捷地安装到目标设备上，无需使用命令行手动操作，提高日常开发调试效率。

**Why this priority**: HAP 包安装是开发者最高频的操作之一，与设备连接并列为最核心的使用场景，直接影响开发效率。

**Independent Test**: 选择本地 HAP 文件，指定目标设备，执行安装，在设备上确认应用已成功安装，即可独立验证。

**Acceptance Scenarios**:

1. **Given** 设备列表中至少有一台已连接的设备，**When** 用户将一个 HAP 文件拖拽到应用窗口内的设备图标区域，**Then** 系统开始将该 HAP 包安装到该设备，并实时显示安装进度。
2. **Given** 用户选择了 HAP 文件和目标设备后正在安装，**When** 安装过程中设备连接断开，**Then** 系统提示「安装失败：设备连接已断开」，并允许用户重新连接后重试。
3. **Given** 用户通过文件选择器选取了一个非 HAP 格式或已损坏的文件，**When** 尝试安装，**Then** 系统在安装前进行校验并提示「文件格式无效」或「安装包不完整」。
4. **Given** 设备已安装过同包名的应用，**When** 用户安装新版本 HAP，**Then** 系统覆盖安装旧版本，并提示「更新安装成功」。
5. **Given** 用户通过拖拽将 HAP 文件放到没有指定设备的应用区域，**When** 仅有一台设备在线，**Then** 系统自动选择该设备进行安装；如果多台设备在线，则提示用户选择目标设备。

---

### User Story 3 - 画面投屏 (Priority: P2)

作为一名 OpenHarmony 应用开发者或演示者，我希望能够将 OpenHarmony 设备的屏幕实时投屏到电脑桌面上，以便进行应用演示、UI 调试或课件录制。

**Why this priority**: 画面投屏是极具价值的可视化功能，但相比设备连接和安装管理来说不是必须的前置功能，可在基础能力完善后独立交付。

**Independent Test**: 选择一台已连接的设备，启动投屏，在电脑窗口中看到设备画面实时同步刷新，且延迟在可接受范围内（<500ms），即可独立验证。

**Acceptance Scenarios**:

1. **Given** 设备列表中有一台已连接的设备且设备屏幕处于亮屏状态，**When** 用户点击「开始投屏」，**Then** 应用打开一个独立的投屏窗口，实时显示设备屏幕画面。
2. **Given** 正在投屏中，**When** 用户在设备上滑动、打开应用等操作，**Then** 投屏窗口中的画面在 500ms 内同步更新。
3. **Given** 投屏窗口已打开，**When** 设备息屏或锁屏，**Then** 投屏画面显示设备屏幕当前状态（锁屏界面），并在设备解锁后自动恢复画面同步。
4. **Given** 正在投屏中，**When** 用户关闭投屏窗口或点击「停止投屏」，**Then** 投屏连接正常断开，设备本身不受任何影响。
5. **Given** 设备通过 Wi-Fi 连接且网络信号较弱，**When** 投屏过程中出现网络波动，**Then** 投屏画面显示当前可传输的帧（不卡死），网络恢复后自动恢复正常帧率。

### User Story 4 - 设备日志查看与导出 (Priority: P3)

作为一名 OpenHarmony 应用开发者，我希望能够在应用内实时查看设备的系统日志和应用日志，并支持按关键字过滤和导出，以便快速定位和排查问题。

**Why this priority**: 日志查看是开发调试的常用辅助功能，能显著提升问题排查效率，但相比设备连接、安装和投屏属于锦上添花的能力。

**Independent Test**: 选择一台已连接的设备，打开日志面板，能看到设备实时输出的日志流，输入关键字后日志被过滤，即可独立验证。

**Acceptance Scenarios**:

1. **Given** 设备列表中有一台已连接的设备，**When** 用户打开「日志」面板，**Then** 系统开始实时显示设备日志输出。
2. **Given** 日志面板正在显示实时日志，**When** 用户在搜索框中输入关键字（如包名或 TAG），**Then** 日志列表仅显示包含该关键字的条目。
3. **Given** 日志面板中有日志数据，**When** 用户点击「暂停/继续」，**Then** 日志滚动暂停或恢复，暂停期间收集的日志在恢复时一并显示。
4. **Given** 日志面板中有日志数据，**When** 用户点击「导出日志」，**Then** 系统将当前可见或已收集的日志导出为文本文件保存到本地。

---

### User Story 5 - 应用偏好设置与 HDC 环境管理 (Priority: P3)

作为一名用户，我希望能够配置应用的偏好设置（如语言、主题、HDC 工具路径），以便应用适应我的使用习惯和工作环境。

**Why this priority**: 设置功能是辅助性的，不影响核心业务流程，但能提升长期使用体验和灵活性。

**Independent Test**: 打开设置页面，修改 HDC 路径或切换主题，修改后立即生效并持久化，即可独立验证。

**Acceptance Scenarios**:

1. **Given** 用户首次启动应用且系统中已安装 HDC 工具，**When** 应用启动，**Then** 系统自动检测系统中 HDC 工具的路径，如检测成功则自动配置；如未检测到则引导用户手动指定路径。
2. **Given** 用户在设置页面中指定了 HDC 工具路径，**When** 点击「测试连接」，**Then** 系统验证该路径下的 HDC 工具是否可用，并反馈验证结果。
3. **Given** 用户在设置中切换了浅色/深色主题，**When** 保存设置，**Then** 应用界面即时切换主题且重新启动后保持该设置。

---

### Edge Cases

- USB 设备在多设备连接场景下，如果存在多台同型号设备，系统通过 SN 序列号区分，确保操作指令发送到正确的设备。
- 设备在投屏或安装过程中突然断连，所有正在进行中的操作立即终止并提示用户，不会出现界面假死。
- HDC 工具本身版本过低或不兼容当前设备系统版本时，系统在首次启动时提示用户升级 HDC 工具。
- 同时向多台设备安装同一个 HAP 包时，每台设备的安装进度独立显示，互不影响。
- 应用窗口最小化到系统托盘时，后台的投屏和日志收集任务继续正常运行（如有必要）。
- 用户在安装或投屏过程中关闭应用，系统弹出确认提示，告知有操作正在进行中。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须能够自动检测通过 USB 连接的 OpenHarmony 设备，并在设备列表中实时反映连接/断开状态。
- **FR-002**: 系统必须支持通过输入 IP 地址和端口号手动连接局域网内的 OpenHarmony 设备。
- **FR-003**: 系统必须展示每台已连接设备的基本信息（设备名称、型号、操作系统版本、SN 序列号、连接方式、电池电量）。
- **FR-004**: 系统必须支持以拖拽方式将 HAP 文件安装到指定设备。
- **FR-005**: 系统必须支持通过文件选择器选取 HAP 文件进行安装。
- **FR-006**: 系统必须在 HAP 安装前进行文件合法性校验（文件格式、完整性）。
- **FR-007**: 系统必须实时显示 HAP 安装进度（百分比或进度条）。
- **FR-008**: 系统必须支持将设备屏幕实时投屏到应用内的独立窗口中显示。
- **FR-009**: 投屏功能必须在设备息屏、亮屏、解锁等状态切换时保持连接并正确显示。
- **FR-010**: 系统必须支持实时查看设备日志输出，并提供按关键字过滤的功能。
- **FR-011**: 系统必须支持日志的暂停、继续和导出为文本文件。
- **FR-012**: 系统必须支持配置 HDC 工具的路径，并提供路径有效性的测试验证功能。
- **FR-013**: 系统必须持久化保存用户的偏好设置（主题、语言、HDC 路径等）。
- **FR-014**: 系统必须在所有长时间操作（安装、投屏、日志采集）中提供明确的进度指示或状态反馈。
- **FR-015**: 系统必须在设备断连时自动终止该设备上的所有进行中操作并向用户发出通知。
- **FR-016**: 系统必须在应用退出时有未完成操作时弹出确认提示，防止用户误关闭。

### Key Entities

- **设备（Device）**: 代表一台通过 HDC 连接的 OpenHarmony 设备。关键属性包括：设备名称、型号、SN 序列号、操作系统版本、连接方式（USB/Wi-Fi）、连接状态（在线/离线）、电池电量、屏幕分辨率。
- **HAP 安装任务（InstallTask）**: 代表一次 HAP 包的安装操作。关键属性包括：目标设备、HAP 文件路径、安装状态（进行中/成功/失败/已取消）、进度百分比、开始时间、结束时间。
- **投屏会话（MirrorSession）**: 代表一次屏幕投屏会话。关键属性包括：目标设备、投屏状态（连接中/投屏中/已断开）、画面分辨率、帧率。
- **日志会话（LogSession）**: 代表一次日志采集会话。关键属性包括：目标设备、过滤关键字、采集状态（收集中/已暂停/已停止）、已收集的日志条目。
- **应用设置（AppSettings）**: 代表用户的偏好配置。关键属性包括：HDC 工具路径、界面主题、显示语言。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从启动应用到看到设备列表的时间不超过 5 秒（假设设备已连接且 HDC 已配置）。
- **SC-002**: USB 设备插入后，系统在 2 秒内完成自动识别并在设备列表中显示。
- **SC-003**: 小型 HAP 包（<50MB）的安装从开始到完成，进度反馈延迟不超过 1 秒。
- **SC-004**: 投屏画面延迟在 USB 连接下不超过 200ms，在 Wi-Fi 连接下不超过 500ms（局域网环境）。
- **SC-005**: 日志面板的日志数据从设备产生到界面显示的延迟不超过 1 秒。
- **SC-006**: 90% 的用户能够在首次使用时无需查阅帮助文档即可完成设备连接和 HAP 安装操作。
- **SC-007**: 应用在多设备同时操作（如同时向 3 台设备安装 HAP）时，界面保持响应流畅，无卡顿。
- **SC-008**: 应用的 CPU 占用率在无投屏任务时不超过 5%，投屏时不超过 30%（中等配置电脑）。

## Assumptions

- 目标用户为 OpenHarmony 应用开发者或测试人员，具备基本的设备调试知识。
- 用户的电脑上已经安装或能够获取 HDC（OpenHarmony Device Connector）工具。
- 设备需要在开发者选项中开启「USB 调试」模式才能被应用检测和管理。
- 网络投屏和设备发现需要电脑与设备处于同一局域网。
- 应用仅支持桌面操作系统（Windows、macOS、Linux），不支持移动端。
- HDC 工具的版本需与 OpenHarmony 设备系统版本兼容，应用不负责 HDC 本身的版本管理。
- 应用默认为单窗口模式，投屏窗口为独立子窗口。
- 界面默认语言为中文，支持后续扩展多语言。
